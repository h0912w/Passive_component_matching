### Major

**M-1: Fraction preserver regex mangles slash-delimited inputs with adjacent numeric tokens**

- Severity: Major
- File: `/src/worker/src/parser.ts`, function `preprocessInput()`, regex on line 83
- Root cause: The regex `(\d+)\/(\d+)(w|W|watt|Watt|WATT|mW|mw)?` matches any digit-slash-digit sequence. In `4.7k/0603/5%/0.25W`, it matches `0603/5` (numerics on both sides of a slash, no W suffix). It appends the default 'W' unit, creating `0603DIV5W`. After slash-to-space and token restoration, this becomes `0603/5W%` — an unrecognized garbage token. Package (0603) and tolerance (5%) are lost.
- The same issue destroys `2K2/1%/1/8W`: matches `2` of `K2` context as `...K2/1` where regex sees `2/1` — actually the token `2K2` is not all digits, but the regex sees the last digit "2" before "/" and "1" after, matching to create `2K2DIV1W%` effectively.
- Reproduction steps:
  1. POST `{"input":"4.7k/0603/5%/0.25W"}` to /parse with debug:true
  2. Observe step02_rule_parsed.data.unmatched_tokens = ["0603/5W%"]
  3. Expected: package = "1608M/0603", tolerance = "5%"
  4. Actual: package = null, tolerance = null
- Impact: Slash delimiter is advertised in the UI hint text and tested in the normal case suite. Any slash-delimited input where two consecutive numeric tokens appear (resistance/package, package/tolerance) will lose those fields.

**M-2: Auto-pass verification when part has no parseable resistance from Description**

- Severity: Major
- File: `/src/worker/src/validator.ts`, function `verifyPart()`, lines 97-98
- Root cause: `else if (extracted.resistance) { resistanceMatch = part.resistance_ohm === null; }` — when the PN's description cannot be parsed to a resistance value, the code considers this a match (true). The status is never set to VERIFICATION_FAILED.
- Combined with M-3, high-stock kΩ resistors whose descriptions fail resistance parsing rank highest and pass verification for any sub-kΩ input.
- Reproduction steps:
  1. POST `{"input":"1R0"}` to /parse
  2. Observe: mouser_pn = "71-RCC04027K50FKED", pn_resistance = null, validation = "PASS"
  3. The returned PN is a 7.5kΩ resistor (description: "7.5Kohms 1% 100ppm")
  4. Expected: PN should be a 1Ω resistor
- Also reproducible with `4.7ohm` → returns 4.75kΩ PN (71-CRCW0603-4.75K-E3)
- Impact: Users receive factually incorrect part numbers. Ordering the returned PN would result in 3-order-of-magnitude wrong resistors.

**M-3: Search keyword lacks unit suffix for sub-kΩ resistor values**

- Severity: Major (contributes to M-2)
- File: `/src/worker/src/mouser.ts`, function `formatResistanceForSearch()`
- Root cause: For values below 1000Ω, the function returns just the numeric string (e.g., "4.7", "1"). The keyword "chip resistor 4.7" matches "4.75Kohms", "14.7Kohms", "4.7Mohms" etc. in Mouser descriptions. High-stock kΩ parts crowd out the correct Ω parts in results.
- Impact: The candidate pool for sub-kΩ searches is polluted, directly enabling M-2.

**m-2: pn_resistance, pn_package, pn_power often null for PASS results**

- Severity: Minor
- For results like `1k 1608 5% 0.25W` (PASS), pn_resistance, pn_package, and pn_power are all null. The Description parser fails to extract these because Mouser descriptions use formats like "CRCW1206 100K 1% 100ppm" where parseResistanceFromDesc's third-priority regex `\b(\d+\.?\d*)\s*(K|M|R)\b` should match "100K" but the word boundary check may fail when followed by " 1%" (space, digit, percent).
- Users see empty PN-side columns in the result table, reducing the value of the comparison view.

**m-3: Inline onclick in src/frontend/index.html vs. event delegation in Blogger build**

- Severity: Minor (maintenance concern)
- frontend/index.html uses `onclick="copySinglePn('...')"` on dynamically created buttons

- The two files have diverged in this regard. If a CSP policy is applied (e.g., Blogger adds one), the frontend version would break.
